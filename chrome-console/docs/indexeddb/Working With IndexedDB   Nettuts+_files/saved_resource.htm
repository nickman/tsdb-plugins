                                                                                                        First off, I think it is important to note the difference between LS (LocalStorage) and ID (IndexedDB - being lazy). Imo, LS is better for storing specific points of atomic data. So for example, your last search, your list of favorite products. Ie, known quantities. ID is better for ad hoc user generated data - a list of notes, a list of TODOs, etc. That explanation may not be very scientific, but it is the rule I go by. Essentially, if it is possible for the user to generate an infinite about the data, I lean towards ID (or WebSQL perhaps in PhoneGap). If the data is set to a certain limit of things, I'll typically use LS.

                                                                                                        Note that LS doesn't have a great way to search, so if searching is important, ID may be better as well.

                                                                                                        Finally - you bring up the point of support. LS is supported a *heck* of a lot more than ID, so that may be a reason to learn that way instead. I'd just try to keep an eye on how much data you push there. Remember that on page load, the browser has to read LS from the file system and that *could* slow things down. LS got kind of a bad rap from some folks, but generally I think the performance concerns were overblown _if_ people are using LS in what I would consider the expected manner.

                                                                                                        Hope this helps - and to be clear - if folks disagree with me I would love to discuss this more. I'm definitely not going to say I have the perfect answer to this.
