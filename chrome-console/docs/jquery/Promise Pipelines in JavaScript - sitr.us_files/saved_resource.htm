                        Promises provide a means to get rid of nested callback in many, but not all, cases. In postWithAuthor the last step, which combines the post and author objects, has to reference both of those objects. But the result of getUser() does not include any information about the post - it just resolves to a user object. The simplest way to reference both objects is via closure scope, which requires nesting the getUser() callback inside of the getPost() callback.

                        The problem with your rewritten version is that the post variable is not in scope in the last callback. If you run that code you will get "ReferenceError: post is not defined" on the line with the $.extend() call.

                        The alternatives to nesting callbacks are to encode all of the information that will be required for subsequent steps into each promise or to update shared variables that are defined outside of the promise callbacks. That second option is not very functional; but it could be done like this:

                        function postWithAuthor(id) {
                        var post;
                        return getPost(id).then(function(p) {
                        post = p;
                        return getUser(post.authorId);
                        }).then(function(author) {
                        return $.extend(post, { author: author });
                        });
                        }

                        I don't think that it is possible to write postWithAuthor in a functional style without nesting callbacks. But the good news is that it is only necessary to nest callbacks when performing sequential steps where one step has to refer back to a detail from the previous step that is not passed forward via promise. You can write code that decorates resolved values at each step to limit nesting to two levels. In fact that is what postWithAuthor is designed to do.
